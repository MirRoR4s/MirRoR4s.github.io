Referrrrer

环境搭建

题目描述：

Defeated the security of the website which implements authentication based on the Referer header.

Format : Hero{flag}
Author : xanhacks

描述让我们击败网站基于 Referer 头搭建的防护措施。

  • 先抓包，重点关注 Referer 头。

访问回显 Hello World，页面无其他有效信息。

题目是给了一个附件的，我们看看源码。

index.js

看起来很简单的样子？get 访问 /admin 然后设置 referer 为对应值就好？

问题并没有这么的简单，明显服务端有一个访问控制。这个访问控制显然不是在代码中设置的，附件中还有一个 nginx 的配置文件。

我对 nginx 相关配置也不是很懂，近期会恶补一下。

果然，在 nginx 的配置文件对 /admin 做了一个访问控制。如果 referer 不是特定域名则直接返回 403。

问题来了，源码要求我们是 “YOU_SHOUD_NOT_PASS!”，而 nginx 要求我们是以上域名，这是一个自相矛盾的事情。

那么我们该如何解出本题呢？且听我娓娓道来。

首先介绍一个有趣的冷知识，在 HTTP 的首部中，用于标识请求来源的首部字段有两个，这两个字段含义、作用一样，但是名字却不一样，分别是 Referer 和 Referrer：

关键的点在于如果我们同时给服务端发送这两个字段会怎么样？这显然就涉及到了一个处理顺序的问题，是先处理 Referer 还是先处理 Referrer 呢？

这就引出了本题的求解方法：

  1. nginx 优先处理 Referer
  2. NodeJS 的 express 库优先处理 Referrer

所以如果我们把 Referer 赋值为 “https://admin\.internal\.com"，把 Referrer 赋值为 “YOU_SHOUD_NOT_PASS!”，这样就可以绕过本题的访问控制机制获得 flag 了。

其实 express 库并不是优先处理 Referrer ，而是对于 Referrer 和 Referer 进行完全一样的处理，二者所执行的代码块完全一样。
