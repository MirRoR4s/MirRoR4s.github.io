Referrrrer

环境搭建

题目描述：

Defeated the security of the website which implements authentication based on the Referer header.

Format : Hero{flag}
Author : xanhacks

描述让我们击败网站基于 Referer 头搭建的防护措施。

  • 先抓包，重点关注 Referer 头。

访问回显 Hello World，页面无其他有效信息。

题目是给了一个附件的，我们看看源码。

index.js

看起来很简单的样子？get 访问 /admin 然后设置 referer 为对应值就好？

问题并没有这么的简单，明显服务端有一个访问控制。这个访问控制显然不是在代码中设置的，附件中还有一个 nginx 的配置文件。

我对 nginx 相关配置也不是很懂，近期会恶补一下。

果然，在 nginx 的配置文件对 /admin 做了一个访问控制。如果 referer 不是特定域名则直接返回 403。

问题来了，源码要求我们是 “YOU_SHOUD_NOT_PASS!”，而 nginx 要求我们是以上域名，这是一个自相矛盾的事情。

那么我们该如何解出本题呢？且听我娓娓道来。

首先介绍一个有趣的冷知识，在 HTTP 的首部中，用于标识请求来源的首部字段有两个，这两个字段含义、作用一样，但是名字却不一样，分别是 Referer 和 Referrer：

关键的点在于如果我们同时给服务端发送这两个字段会怎么样？这显然就涉及到了一个处理顺序的问题，是先处理 Referer 还是先处理 Referrer 呢？

这就引出了本题的求解方法：

  1. nginx 优先处理 Referer
  2. NodeJS 的 express 库优先处理 Referrer

所以如果我们把 Referer 赋值为 “https://admin\.internal\.com"，把 Referrer 赋值为 “YOU_SHOUD_NOT_PASS!”，这样就可以绕过本题的访问控制机制获得 flag 了。

其实 express 库并不是优先处理 Referrer ，而是对于 Referrer 和 Referer 进行完全一样的处理，二者所执行的代码块完全一样。

Crypto 复现

Hyper Loop

可以用式子形式化地表达一下加密过程，不妨设 flag 的每一位是 $m_i$，最终的输出的每一位是 $c_i$。

我们知道，每一轮都会生成一个 18 字节的密钥，由 3 个完全相同的 6 字节的子密钥组成。不妨设为 $K_{ij}$ ，其中$0\leq,i\leq31,0\leq,j\leq,17$，$K_{ij}$ 表示第 i 轮生成的密钥的第 j 个字节。

所以
$$
\begin{align}
c_0=m_0\oplus,K_{0,0}\oplus,K_{1,0},…\oplus,K_{31,0}\\
c_1=m_1\oplus,K_{0,1}\oplus,K_{1,1},…\oplus,K_{31,1} \\
… \\
c_{17}=m_{17}\oplus,K_{0,17}\oplus,K_{1,17},…\oplus,K_{31,17} \\
=m_{17}\oplus,K_{0,5}\oplus,K_{1,5},…\oplus,K_{31,5}\\
\end{align}
$$

上式每个 $m_i$ 右边一大串 $K_{i,j}$ 的异或结果可以看成一个值，不妨设为 $k_i$，其中 $0\leq,i\leq17$

那么我们有：
$$
c_0=m_0\oplus,k_0\
c_1=m_1\oplus,k_1\
c_5=m_5\oplus,k_5\
c_6=m_6\oplus,k_6
$$
利用 $c_{17}$ 和 $m_{17}$ 求出 $k_5$，从而求出 $m_5$
$$
\begin{align}
m_6 &=c_6\oplus,k_6\
k_6=k_0=c_0\oplus,m_0\
m_6&=c_6\oplus,c_0\oplus,m_0
\end{align}
$$

有意思的地方在于我们已经知道了明文的前五个字符是 “Hero{“，最后一个字符是 “}”。那么根据异或运算的特点，可以恢复 $k_i$，其中 $i=0,1,2,3,4,17$。我们知道 $k_i$ 和 $k_{i+6}$ 是相等的，这是解题的关键。解密出 $k_0,k_1,k_2,…k_5$ ，就解出了 $k_6,k_7,…k_{11}$

Lossy

题目地址：https://github.com/HeroCTF/HeroCTF_v5/tree/main/Crypto/Lossy

ECB 模式 AES 算法加密明文。已经给出了密钥，应该可以解密。

但是我发现将 ct 转为 bytes 的时候报错了，

两个十六进制才能转成一个字节，我们这里是63个十六进制字符串，最后一个似乎转不了。

问题应该还是出在 to_hex 函数上，它是如何产生的呢？

原因在于 hex() 函数对于 0x00 - 0x0f 之间的值编码长度仅有 3，hex 不会在高位填充前置零，而大于这个区间的值编码长度是 4。也就是说，本题给出的 key 和 ct 少了一些前置的零，我们需要找到这些前置零。如何找呢？由于 key 和 ct 的长度都不算太长，而且缺少的前置零的个数也不太多，所以可以通过爆破来做。

这其实就是一个组合的问题，比如本题的 key 长度是 29，那么我们就知道 key 少了 3 个前置零。那么在这 29 个位置中必定是有 3 个位置少了前置零的，总共的可能性是 $C_{29}^3$，难度不大。
