二. Repeater

2.1 什么是 Repeater？

在我们开始使用 Repeater 之前，了解它的作用会很有帮助。

简而言之：Burp Suite Repeater 允许我们随意制作和/或将截获的请求中继到目标。 通俗地说，这意味着我们可以获取在 Proxy 中捕获的请求，对其进行编辑，并根据需要多次重复发送相同的请求。 或者，我们可以手动创建请求，就像我们从 CLI（命令行界面）中所做的那样，使用诸如 cURL 之类的工具来构建和发送请求。

这种多次编辑和重新发送相同请求的能力使 Repeater 非常适合在端点进行任何类型的手动探索，为我们提供了一个漂亮的图形用户界面 (GUI) 来编写请求有效负载和大量视图（包括用于响应的图形视图），以便我们可以看到我们手工操作的结果。

Repeater 界面可以分为六个主要部分——可以在以下要点下方找到带注释的图表：

  • 在选项卡的最左上角，我们有一个 Repeater 请求列表。 我们可以有许多不同的请求通过 Repeater：每次我们向 Repeater 发送新请求时，它都会出现在这里。
  • 在请求列表的正下方，我们有当前请求的控件。 这些允许我们发送请求、取消挂起的请求以及在请求历史记录中前进/后退。
  • 仍然在选项卡的左侧，但占据了大部分窗口，我们有请求和响应视图。 我们在请求视图中编辑请求，然后按发送。 响应将显示在 Response 视图中。
  • 在请求/响应部分的上方，右侧是一组选项，允许我们更改请求和响应视图的布局。 默认情况下，这通常是并排的（水平布局，如屏幕截图所示）； 但是，我们也可以选择将它们放在彼此的上方/下方（垂直布局）或放在单独的选项卡中（组合视图）。
  • 在窗口的右侧，我们有检查器，它允许我们将请求分开，以比使用原始编辑器更直观的方式分析和编辑它们。 我们将在稍后的任务中介绍这一点。
  • 最后，在 Inspector 上方我们有我们的目标。 很简单，这是我们向其发送请求的 IP 地址或域。 当我们从 Burp Suite 的其他部分向 Repeater 发送请求时，这将被自动填写。

如果现在这还不太有意义，请不要担心——在接下来的任务中，您将有很多机会亲身了解它的作用！

2.3 View

Repeater 为我们提供了多种方式来呈现对我们请求的响应——这些方式从十六进制输出一直到页面的完全渲染版本。

我们可以通过查看响应框上方来查看可用选项：

我们这里有四个显示选项：

  • Pretty：这是默认选项。 它采用原始响应并尝试稍微美化它，使其更易于阅读。
  • Raw：来自服务器的纯净、未美化的响应。
  • Hex：此视图采用原始响应并为我们提供了它的字节视图——如果响应是二进制文件，则特别有用。
  • Render：呈现视图呈现页面，就像它在浏览器中显示的那样。 考虑到我们在使用 Repeater 时通常会对源代码感兴趣，因此虽然用处不大，但这仍然是一个巧妙的技巧。

在大多数情况下，“Pretty”选项就足够了； 但是，仍然值得了解如何使用其他三个选项。

视图按钮的右侧是 “Show non-printable characters 显示不可打印字符” 按钮 (\n)。 这个按钮允许我们显示通常不会出现在 Pretty 或 Raw 视图中的字符。 例如，响应中的每一行都将以字符 \r\n 结尾————这些字符表示一个回车符后跟一个换行符，并且是 HTTP 标头解释方式的一部分。

虽然大多数任务不需要，但此选项仍然可以派上用场。

2.4 Inspector

在许多方面，Inspector 完全是对 Repeater 窗口的请求和响应字段的补充。 如果您了解如何读取和编辑 HTTP 请求，那么您可能会发现您根本很少使用 Inspector。

也就是说，这是获得请求和响应的美化分解的极好方法，以及通过实验了解使用更高级别的 Inspector 所做的更改如何影响等效的原始版本。

Inspector 可以在 Proxy 和 Repeater 中使用。 在这两种情况下，它都出现在窗口的最右侧，并为我们提供了请求和响应中的组件列表：

其中，请求部分几乎总是可以更改，允许我们添加、编辑和删除项目。 例如，在 Request Attributes 部分，我们可以编辑请求中处理位置、方法和协议的部分； 例如更改我们要检索的资源，将请求从 GET 更改为另一种 HTTP 方法，或将协议从 HTTP/1 切换到 HTTP/2：

可用于查看和/或编辑的其他部分是：

  • Query Parameters 查询参数，指的是在 URL 中发送到服务器的数据。 例如，在对 https://admin.tryhackme.com/?redirect=false 的 GET 请求中，有一个名为“redirect”的查询参数，其值为“false”。
  • Body Parameters 正文参数，与查询参数做同样的事情，但用于 POST 请求。 我们在 POST 请求中作为数据发送的任何内容都将显示在此部分中，再次允许我们在重新发送之前修改参数。
  • 如您所料，Request Cookies 请求 Cookie 包含随每个请求一起发送的 cookie 的可修改列表。
  • Request Headers 请求标头允许我们查看、访问和修改（包括完全添加或删除）与我们的请求一起发送的任何标头。 尝试查看 Web 服务器将如何响应意外标头时，编辑这些内容非常有用。
  • Response Headers 向我们显示服务器响应我们的请求发回的标头。 这些无法编辑（因为我们无法控制服务器返回给我们的标头！）。 请注意，此部分只会在我们发送请求并收到响应后显示。

这些组件都可以在请求和响应部分中以文本形式找到； 但是，很高兴以 Inspector 提供的表格格式查看它们。 在 Inspector 中添加、删除和编辑标头是非常值得的，以便在您这样做时感受原始版本的变化。
