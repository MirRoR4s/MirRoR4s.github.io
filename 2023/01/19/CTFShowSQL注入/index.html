试试直接读取文件 /flag，但是发现并不在这里。但是经过测试是可以读取的，因为我读到了当前的 php 文件

后面发现有一个 ctfshow_user2 表，flag 就在这个表里

web173#

和上题几乎一模一样，但是 flag 又换了一个表。同时这是一个有过滤的字符型注入，添加了检查结果中是否匹配正则表达式 /flag/i ，

针对这种情况，我们可以将查询结果进行编码，mysql 中常见的编码函数有

  • to_base64
  • hex

添加hex函数绕过正则过滤

web174#

和前几题的查询语句一样，但是在回显的时候不能出现 flag 字符和数字

不幸地是，这两种编码都会包含到数字，所以这题我们不能采用联合注入了，要使用盲注。注意查询的时候访问的是 v4.php

web175#

平台说这是最后一个无过滤的 sql 注入~ 但是为什么回显还是有过滤？？

过滤了 ascii 0-127,相当于没有回显，所以改上题的布尔盲注为时间盲注，同样查询语句是在 v5.php！

以下题目开始过滤啦

前言#

以下题目大都过滤了空格，一般代替空格的字符串有

  • %09
  • %0a
  • %0b
  • %0d
  • %0c

web176#

这里其实是对 select 进行了过滤，大小写就能绕过。但是也可以如下拿 flag

web177#

这里应该是过滤掉了空格，只要存在空格就把整个语句都清空。用 %09 替代空格即可

踩坑点：

使用 –+ 作为注释符会失效，因为最后的 + 在url解码后被识别为空格，然后触发了题目的 waf

所以可以将 # 编码为 %23 来作为注释符，不编码的话 # 在url中有特殊含义会有些问题

web178#

依旧过滤了空格，上一题的答案还能用。不知道为什么这样的语句行不通，所以具体过滤了什么呢？

但是上题答案可以

web179#

依旧是过滤了空格，但是现在 %09 不好用啦。但是可以用 %0c 代替空格

web180#

这一关过滤了注释符 %23,而且空格还是只能用 %0c 代替。那我们只能使用另一个注释符–空格了，将空格替换成 %0c 就好

一开始是想在联合查询的最后一个字符左边加上单引号，然后和题目的单引号形成一个完整的单引号以此绕过过滤的。

但是发现 group_conat 语句只能在联合查询的最后一个位置，不知道为什么
