环境搭建#

去github页面下载源码然后用 idea 打开即可。同时使用官方给的命令把容器搭建起来：

代码审计#

JWT Token 猜解实验的目标是改token为管理员的token并且重置投票。

实验思路：通过抓取 “重置投票” 的 HTTP 请求数据包，以期在找到关键的接口信息后进行定向的代码审计

首先是初始页面（老版本 WebGoat 的 JWT Token 实验应该是在 A2，我们现在是在 A7）

投票页面在第五关

当点击 reset vote 按钮（下图的垃圾桶图标）之后会向 JWT/Votings 发送一个 http 请求包

为了在源码中快速定位到该接口对应的方法，可以在idea下输入 ctrl+shift+r 查找 Votings

注意正确的代码在最后一条结果，不要选到 test 的代码了!

跟进去发现如下：

首先有一个判断 accesstoekn 是否为空的操作，为空直接退出来。

圈红的代码顾名思义一下就是设置 jwt 的签名密钥之后去解析 accessToken，然后获取 claims 中的 admin 参数的值，若 “Boolean.valueOf(String.valueOf(claims.get(“admin”)))” 的返回值为 true，则判断该 token 是有效的，并将进行“重置投票”操作。

那么变量 accessToken 是如何产生的？通过 resetVotes 方法的注解，可以发现该变量存储于 Cookie 中，且 Cookie 的键名为 access_token

那这个 Cookie 对象在哪里被创建和定义呢？这里我也不知道咋找的，书里直接说是在 login 方法中被创建。可能用户在登录的时候会进行 Cookie 相关设置吧。idea下输入 ctrl+f 搜 login 找到了这个方法

在 134 行 用 access_token 和 token 参数的值实例化了一个 Cookie 类对象（这里可能是建立了一个键值对，键是 access_token，值是 token），而这个 token 显然就是我们在重置投票接口那里的 accesstoken。现在我们终于弄清楚这个 accesstoken 是咋来的了，那我们到底如何进行漏洞利用呢？

根据前面的分析我们知道 jwt 的签名密钥是一个固定值，这使得我们可以伪造 jwt Tokens！

我们要做的事情就是访问 /JWT/Votings 接口然后伪造管理员的 accesstoken。我们先用一个普通用户的 accsstoken 访问过去看看回显
