knock-knock-密码学函数使用不当#

题目描述#

Knock knock? Who’s there? Another pastebin!!

题目内容#

本题是根据id和token访问对应的笔记。阅读源代码会发现flag应是在id=0的位置，但是我们不知道这个位置对应的token是什么。token的生成是 sha256和随机密钥的消息认证码，在不知道密钥的前提下是无法获得token的。

但是如果你足够细心，会发现 Crypto.randomUUID 在调用的时候没有带括号，这意味着 密钥的生成是固定的。

可以看到 UUid 函数未带括号的时候密钥是一个未定义的状态！所以我们在本地用id=0运行一遍就可以得到和flag相同的token了，如果node版本一致的话。题目的node版本是FROM node:17.4.0-buster-slim

可以看到 mock函数的返回结果被插入到 id 为 result的元素中，这很显然是一个 XSS的点 ！继续看看mock函数

经过以上一通分析，我们知道我们可以在 demo参数输入数据并将其传入到 blazingfast.mock函数进行处理，而 这个函数正是来自于 blazingfast.c。具体可以看一下这个：https://zhuanlan.zhihu.com/p/68048524

看看 c 代码吧：

可恶啊，传入 mock 函数的 字符串 不可以有大于号、小于号、于号、双引号。我们的 XSS payload遭受了挑战。从这篇文章获得灵感：https://jlajara.gitlab.io/XSS_20_characters

注意上图不是两个 ff，是一个长的很像两个 ff 的单个 Unicode字符。妙处发生在浏览器在解释这个字符的时候，会将其扩展为两个 ff，这就导致长度增加了（由1到2）。如果是 Unicode 字符 ffi，则长度由1变到3

从上图可以看到明明是长度为 1 的字符串，但是最后居然能获得三个 Unicode 码值。这种差异就允许我们绕过XSS的黑名单检查，因为其 length 值是初始传入的长度。

我们可以填充特别多的 ffi，然后其后面插入我们的XSS payload 进行攻击。

但是直接传入会爆出一个 ALERT 未定义的错误，这是因为我们的输入在处理过程中被传成大写了。

可以通过将大小写敏感的函数进行 html 编码解决。经过我的测试，只需要将函数名编码即可。单引号、左右括号、点号都是不需要编码的。
